# 01 — Churn CI/CD: Training Pipeline + Local Deployment

This lesson turns "a script that trains a model" into the two halves of a real ML delivery system: a **layered training pipeline** that produces a versioned artifact behind a **metric quality gate**, and a **local serving process** that exposes that artifact over HTTP. Then it wires both into a **GitHub Actions pipeline**: `test → train (gate) → smoke-test`, with the model traveling between jobs as an artifact.

No monitoring yet — the model goes live blind. Seeing why that is a problem is Lesson 02's job.

---

## 1. Objectives

After this lesson you can:

- Structure a training pipeline in layers (`data / features / model / serving / pipeline`) where every module has one job and receives its collaborators by injection.
- Explain what a metric gate is, where it sits, and trace the exact mechanism by which a weak model turns into a red CI run with no published artifact.
- Explain why tests run **before** training (CI before CT) and what each test guards.
- Deploy a trained artifact locally: load it once, serve it behind validated endpoints, and verify it with real requests.
- Read and write the full GitHub Actions workflow, including the artifact handoff between jobs.
- Reproduce every step by hand and validate it against `expected_outputs/`.

## 2. Theory

### 2.1 The dataset: synthetic, deterministic, committed

`data/churn.csv` — 2,000 customers of a fictional subscription business, generated by `shared/churn_data.py` with a fixed seed (42).

| Column | Type | Generated as | Role |
|---|---|---|---|
| `tenure_months` | int | uniform 1–72 | numeric feature |
| `monthly_charges` | float | uniform 20–120, 2 decimals | numeric feature |
| `total_charges` | float | `tenure_months × monthly_charges` | numeric feature (deliberately collinear — realistic) |
| `contract_type` | str | 50% `month_to_month`, 30% `one_year`, 20% `two_year` | categorical feature |
| `payment_method` | str | 40% `credit_card`, 30% `bank_transfer`, 30% `electronic_check` | categorical feature |
| `support_tickets` | int | Poisson(1.5) | numeric feature |
| `churn` | int | Bernoulli(sigmoid(churn_logit)) | target |

The label is sampled from a known formula — you can reason about *why* the model behaves as it does:

```
churn_logit = -0.5
            + 2.2   · (contract_type == month_to_month)
            + 1.1   · (payment_method == electronic_check)
            - 0.09  · tenure_months
            + 0.022 · (monthly_charges − 70)
            + 0.55  · support_tickets
```

High risk ⇒ new customer, month-to-month, paying by electronic check, opening tickets. That is exactly the profile of request 1 in the walkthrough — and why it scores 0.9966.

Why synthetic and committed? Three MLOps reasons, not modeling reasons:

1. **Reproducibility is testable.** Same seed ⇒ byte-identical CSV. You will prove this with a checksum in the walkthrough.
2. **Expected outputs can be exact.** Deterministic data + fixed seeds + locked dependency versions (`uv.lock`) ⇒ the metrics in this README are the metrics you get.
3. **The pipeline, not the data, is the subject.** Lessons swap in new data later (that is drift, Lesson 03); the machinery must already be trustworthy.

### 2.2 From script to layered pipeline

The reference implementation could be one 50-line script. It is instead five layers, because every MLOps capability you add later — monitoring, drift detection, registries, deployment strategies — attaches to **one** layer without touching the others.

| Layer | Modules | Single responsibility |
|---|---|---|
| `data` | `loader`, `schema_validator`, `splitter` | get a frame in, refuse bad frames, split it |
| `features` | `encoder` | declare how raw columns become model inputs |
| `model` | `trainer`, `evaluator`, `metric_gate`, `persistence` | fit, measure, gate, save/load |
| `serving` | `schemas`, `app`, `run_server` | expose a saved artifact over HTTP |
| `pipeline` | `config`, `train_pipeline` | **composition root**: wire concrete pieces, in order |

Two dependency rules hold everywhere:

1. **Only `pipeline` (and `run_server`, serving's own composition root) instantiate things.** Everything else receives collaborators as parameters. `train_model_pipeline` takes the classifier; it never builds a `LogisticRegression`. That is why `tests/test_predict_endpoint.py` can inject a 12-row toy model, and why Lesson 06 will swap persistence for a model registry without touching training logic.
2. **Inner layers never import outward.** `model` knows nothing about `pipeline`; `serving` touches only `model.persistence` (to load) and its own `schemas`. Check the imports — they are the architecture.

### 2.3 The metric gate is the contract between training and shipping

A gate is a threshold check placed **after evaluation, before anything is packaged or published**:

```python
if observed_value < minimum_value:
    raise ValueError(...)
```

Trace the full mechanism, because this one chain is the heart of the lesson:

```
roc_auc < 0.85
  → enforce_metric_gate raises ValueError
  → the training process dies with a traceback, BEFORE save_model_pipeline runs
  → exit code 1
  → the `train` job step fails
  → the job turns red, `smoke-test` (needs: train) is skipped
  → upload-artifact never executes → no artifact exists to deploy
```

No `try/except` anywhere. The *absence* of error handling is the design: an unhandled exception is precisely how a Python process tells CI "do not ship this". The gate threshold (`roc_auc ≥ 0.85`) lives in `pipeline/config.py` next to the seed and paths — it is configuration, not logic.

ROC-AUC gates instead of accuracy because the dataset is imbalanced (~31% churn): always predicting "no churn" scores 0.69 accuracy with zero usefulness, while ROC-AUC measures ranking quality independent of the 0.5 cut.

### 2.4 CI before CT: what the tests guard

The `test` job costs seconds; the `train` job costs minutes (real projects: hours). Order is economics — never spend training compute on code that cannot pass its tests. What each file guards:

- `test_schema_validator.py` — the pipeline refuses malformed data (missing columns, NaNs, non-binary target) *before* training on it.
- `test_metric_gate.py` — the gate fails below threshold, passes at exactly the threshold, and rejects metric names that do not exist (a typo in config must not silently disable the gate).
- `test_predict_endpoint.py` — the serving contract: response shape, probability/label consistency, and 422 rejection of unknown categories. Runs against an injected toy model — no artifact needed, which is what allows `test` to run before `train` exists.

### 2.5 Local deployment = a saved artifact + a serving process

"Deploy" in this lesson means: a **separate process** loads `artifacts/model.joblib` **once at startup** and answers HTTP requests. Training and serving never share memory — they share a file. Locally the handoff is the filesystem; in CI it is the uploaded/downloaded artifact; in Lesson 04 it becomes a container image. The pattern is identical, only the transport changes.

Two details carry most of the engineering value:

- **The artifact is encoder + classifier fused** (one sklearn `Pipeline`). Serving cannot accidentally scale or encode differently than training did — the #1 classic train/serve skew bug is structurally impossible.
- **Validation is symmetric.** Training validates frames at its boundary (`schema_validator`); serving validates requests at its boundary (pydantic `Literal` types in `schemas.py`). Request 3 in the walkthrough proves bad categories die at the API edge with a 422 — the model is never called.

### 3.1 Training data flow through the layers

```
data/churn.csv
      │
      ▼
┌────────────────────────────────┐
│ data layer                     │
│   load_churn_frame             │  missing file / empty file ──► ValueError
│   validate_required_columns    │

│   validate_binary_target       │
│   split_features_and_target    │
│   split_train_test             │  stratified, seed=42, 80/20
└───────────────┬────────────────┘
                │ 1600 train rows · 400 test rows
                ▼
┌────────────────────────────────┐
│ features layer                 │
│   build_feature_encoder        │  StandardScaler(4 numeric) + OneHotEncoder(2 categorical)
└───────────────┬────────────────┘
                │ unfitted ColumnTransformer
                ▼
┌────────────────────────────────┐
│ model layer                    │
│   build_model_pipeline         │  encoder + LogisticRegression  (classifier is INJECTED)
│   train_model_pipeline         │
│   evaluate_classifier          │  accuracy · precision · recall · roc_auc
│   enforce_metric_gate          │  roc_auc < 0.85 ──► ValueError, NOTHING is saved
│   save_model_pipeline          │
│   save_metrics                 │
└───────────────┬────────────────┘
                ▼
artifacts/model.joblib  +  artifacts/metrics.json
```

The `pipeline` layer is invisible in the data flow on purpose: it contributes no transformation, only the wiring above, read top to bottom in `train_pipeline.py`.

```
client                    uvicorn (run_server)         FastAPI app (create_application)

  │                              │                       captured by the endpoint via closure
```

### Step 5 — make the gate fail on purpose

The config is a frozen dataclass, so an impossible threshold can be injected for one run without editing any file:

```bash
uv run python -c "
from dataclasses import replace
from src.pipeline.config import DEFAULT_TRAINING_CONFIG
from src.pipeline.train_pipeline import run_training_pipeline
run_training_pipeline(replace(DEFAULT_TRAINING_CONFIG, gate_minimum_value=0.99))
"
echo "exit code: $?"
```

Expected output: the same log up to `test metrics: ...`, then a traceback ending in (`expected_outputs/gate_failure_log.txt`):

```
ValueError: metric gate failed: roc_auc=0.9222 is below minimum 0.99
exit code: 1
```

That `exit code: 1` **is** the CI integration. GitHub Actions has no idea what ROC-AUC is — it only sees a step exit non-zero and turns the job red. Note also what is *absent*: no `artifacts written` line. The gate killed the run before persistence.

### Step 7 — read the CI pipeline, then watch it run

Open `.github/workflows/01-churn-ci.yml` and map every block to lesson 00 §2.5 vocabulary:

- `on:` — `push` to `main` and `pull_request`, both filtered with `paths:` so this pipeline only spends compute when this lesson's code, `shared/`, the dependency files, or the workflow itself change. Plus `workflow_dispatch` for a manual button.
- `defaults.run.working-directory` — every `run:` step executes from the lesson folder, exactly like your terminal in this walkthrough. (`uses:` steps ignore it — that is why artifact paths are repo-relative.)
- Job `test` → `pytest -q` — must print `11 passed`.
- Job `train` (`needs: test`) — runs the same module you ran in step 4; the gate is *inside* the module, so CI inherits it for free; uploads `artifacts/` as artifact `churn-model`.
- Job `smoke-test` (`needs: train`) — downloads `churn-model`, starts the server in the background, curls `/health` with retries until the model is loaded, then sends the high-risk `/predict` request. Any failed curl (`--fail`) fails the job.

Push a change touching this lesson (or open a PR), then compare the Actions run against `expected_outputs/workflow_run_summary.txt`: three green jobs in sequence and a `churn-model` artifact with 2 files on the run page.

## Lesson summary

- A training pipeline is layers wired by one composition root: data validation and splitting, an injected encoder+classifier fused into a single artifact, evaluation, and a metric gate that turns a weak model into a non-zero exit code — which is all CI needs to block shipping.
- Local deployment is a separate process that loads the published artifact once and serves it behind schema-validated endpoints; training and serving share a file, never memory.
- The GitHub Actions workflow is the same three things you ran by hand — pytest, the training module, server + curl — as three chained jobs with the model traveling between them as an artifact.

**Next — Lesson 02 (Monitoring):** the model is now live and completely blind: you cannot see latency, error rates, prediction distributions, or whether performance is decaying. Lesson 02 adds service metrics + model metrics, structured logging, and a simple dashboard — the eyes that Lesson 03's drift detection will need.
